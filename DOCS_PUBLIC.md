## 1. Описание продукта

**MC Panel** — корпоративная веб-платформа для централизованного администрирования распределённого кластера игровых серверов Minecraft, эксплуатируемого компанией ООО «Биг Девелопмент».

Продукт решает задачи операционного управления игровой инфраструктурой:
- Мониторинг состояния серверов в режиме реального времени
- Управление жизненным циклом игровых экземпляров
- Удалённое исполнение административных команд
- Управление пользователями и аудит действий персонала
- Аналитика игровой активности

## 4. Технологический стек

### 4.1 Серверная часть

- **Платформа исполнения:** Node.js LTS
- **HTTP-фреймворк:** Express
- **СУБД:** Реляционная встроенная база данных
- **Аутентификация:** JWT + bcrypt
- **Реалтайм-протокол:** WebSocket
- **Интеграция с игровыми серверами:** RCON (Remote Console Protocol)
- **Мониторинг доступности:** Server List Ping (SLP)
- **Управление контейнерами:** Docker CLI

### 4.2 Клиентская часть

- **UI-библиотека:** React 18
- **Сборщик:** Vite
- **Маршрутизация:** React Router
- **Стилизация:** TailwindCSS (utility-first подход)
- **HTTP-клиент:** Axios
- **Графики:** Собственная реализация на SVG

### 4.3 Обоснование выбора технологий

| Решение | Альтернативы | Обоснование |
|---|---|---|
| Node.js | Go, Java, Python | I/O-bound характер задачи (множество ожиданий ответа от внешних систем) идеально подходит под асинхронную модель Event Loop |
| Встроенная SQLite | PostgreSQL, MySQL | Нет необходимости в отдельном сервере БД; объём данных не превышает 10 GB; отсутствие нативных бинарных зависимостей упрощает развёртывание |
| React + Vite | Vue, Angular, Svelte | Наибольшая экосистема компонентов, низкий порог входа для расширения команды, быстрая сборка через esbuild |
| TailwindCSS | Bootstrap, Material UI | Минимизация итогового размера CSS-бандла за счёт purge неиспользуемых классов; единый дизайн-язык без CSS-конфликтов |
| Собственные SVG-графики | Recharts, Chart.js, D3 | Полный контроль над визуальным стилем под брендинг компании; меньший размер бандла; отсутствие внешних зависимостей |

## 5. Архитектурные принципы

### 5.1 Разделение ответственности (Separation of Concerns)

Серверная часть разделена на три слоя:
- **Маршрутизация** — приём HTTP-запросов, валидация входных параметров, делегирование
- **Сервисный слой** — бизнес-логика, не зависящая от транспорта
- **Слой доступа к данным** — работа с СУБД и внешними системами

Это обеспечивает тестируемость и возможность переиспользования логики.

### 5.2 Контрактно-ориентированный API

REST API спроектирован по принципам REST с явной семантикой HTTP-методов:
- GET — чтение без побочных эффектов
- POST — создание ресурса или выполнение действия
- PUT / PATCH — изменение ресурса
- DELETE — удаление

Все ответы возвращают JSON фиксированного формата с явным разделением успешных и ошибочных состояний через HTTP-коды.

### 5.3 Stateless-сервер

Серверная часть не хранит пользовательских данных в памяти процесса. Состояние сессии передаётся через JWT в httpOnly-cookie, что позволяет масштабировать сервер горизонтально без необходимости sticky-сессий на балансировщике.

### 5.4 Двухуровневый цикл мониторинга

Для оптимизации нагрузки на инфраструктуру реализован двухуровневый цикл опроса:
- **Быстрый цикл (5 секунд)** — лёгкие операции (ping серверов), результаты транслируются клиентам через WebSocket
- **Медленный цикл (30 секунд)** — тяжёлые операции (RCON, запись в БД)

Это даёт пользователю ощущение реалтайма при минимальной фактической нагрузке.

## 6. Модель данных

База данных содержит следующие основные сущности:

- **Пользователи** — учётные записи операторов панели
- **Сессии** — активные сеансы авторизации с возможностью отзыва
- **История онлайна** — снимки игровой активности для построения аналитики
- **Игровые сессии** — отслеживание входов и выходов игроков на серверах
- **Журнал команд** — все команды, выполненные через RCON-консоль
- **Журнал аудита** — все мутирующие действия операторов

Все запросы к БД параметризованы через prepared statements, что исключает возможность SQL-инъекций.

## 7. Безопасность

Реализованы следующие меры безопасности:

| Угроза | Защита |
|---|---|
| Кража учётных данных | bcrypt-хэширование с раундом 10 |
| Brute-force атаки | Ограничение скорости попыток входа |
| XSS-атаки | httpOnly-cookies для хранения токенов |
| CSRF-атаки | SameSite=Lax cookies |
| SQL-инъекции | Параметризованные запросы |
| Перехват сессии | SHA-256 хэширование токенов в БД |
| Несанкционированный доступ | Трёхуровневая ролевая модель + middleware-проверки |
| Расследование инцидентов | Полный журнал аудита с IP-адресами |

## 8. Подсистемы

### 8.1 Подсистема авторизации

Реализует:
- Аутентификацию по логину и паролю
- Выдачу JWT-токенов со сроком действия 12 часов
- Разграничение доступа на уровне маршрутов и операций
- Принудительный отзыв сессий

### 8.2 Подсистема мониторинга

- Опрос статуса игровых серверов через SLP
- Получение TPS, MSPT и метрик памяти через RCON
- Агрегация данных с прокси-сервера и backend-серверов
- Сохранение исторических данных в БД

### 8.3 Подсистема управления контейнерами

Обёртка над Docker CLI с поддержкой:
- Запуска, остановки, перезапуска
- Graceful shutdown через предварительную команду RCON `stop`
- Возврата состояния контейнера

### 8.4 Подсистема реального времени

WebSocket-сервер с:
- Аутентификацией через httpOnly-cookie на этапе HTTP-upgrade
- Heartbeat для определения мёртвых соединений
- Push-уведомлениями всем подключённым клиентам

### 8.5 Подсистема журналирования

- Каждое значимое действие пишется в журнал аудита
- Журнал доступен только пользователям с ролью super-admin
- Поддерживается фильтрация по типу действия, пользователю, периоду

## 9. Возможности расширения

Архитектура спроектирована с учётом дальнейшего масштабирования:

- **Распределённая инфраструктура** — переход к агентной модели с агентами на каждом физическом узле
- **Миграция СУБД** — стандартный SQL без специфичных конструкций позволяет перейти на PostgreSQL без переписывания запросов
- **Кластеризация панели** — stateless-архитектура позволяет запускать несколько экземпляров за балансировщиком
- **Интеграция с системами мониторинга** — возможность добавления Prometheus-экспортёра

## 10. Экономический эффект

Разработка собственного решения вместо использования open-source альтернатив (Pterodactyl, PufferPanel) обоснована следующими факторами:

- **Снижение TCO** — отсутствие необходимости в выделенном MySQL-сервере, Redis-кэше, отдельном Daemon-процессе на Java
- **Контроль над данными** — отсутствие зависимости от стороннего ПО снижает риски в условиях санкционных ограничений
- **Защита коммерческой тайны** — собственный код не передаёт информацию о внутренней инфраструктуре во внешние системы
- **Брендирование интерфейса** — соответствие фирменному стилю компании
- **Гибкость доработок** — отсутствие зависимости от чужого release-цикла и архитектурных решений
